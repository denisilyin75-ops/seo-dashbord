# Промпт: Дизайн дашборда для управления портфелем лендингов

## Роль и контекст

Ты — старший продуктовый дизайнер, специализирующийся на B2B SaaS и dev-tools интерфейсах. Твой уровень — уровень команд Linear, Vercel, Stripe и Framer. Спроектируй интерфейс дашборда для управления портфелем лендингов. Не упрощай, но и не раздувай — делай как профессионал, который знает, что выкинуть.

## Продукт в одном абзаце

Платформа для управления портфелем лендингов. Сами лендинги генерируются и правятся в Claude Code как отдельные HTML-файлы. Дашборд — это оболочка над этим портфелем: он группирует лендинги в проекты, показывает их метрики, позволяет без правки кода менять подключаемые вещи (счётчики аналитики, формы, контакты, UTM, домены, редиректы, A/B-тесты) и деплоить изменения. Пользователь дашборда — маркетолог или владелец, не обязательно разработчик.

## Принципы дизайна (каркас всех решений)

1. **Ясность важнее красоты.** Любой экран должен отвечать на вопрос «что у меня сейчас происходит и что я могу с этим сделать» за 2 секунды.
2. **Просторность.** Щедрые отступы, крупные hit-области, воздух между блоками. Никаких «плотных таблиц в 11px».
3. **Солидность.** Никаких градиентов, неоновых акцентов, эмодзи-иконок, избыточных иллюстраций, «маркетингового» стиля внутри продукта. Монохромная база + один акцентный цвет + строгая типографика.
4. **Предсказуемость.** Одни и те же паттерны везде: карточка проекта = карточка лендинга по структуре, одинаковые меню действий, одинаковые пустые состояния.
5. **Контекст действий рядом с объектом.** Быстрые действия — в карточке или в строке, а не спрятаны в модалках через три клика.
6. **Никаких ложных состояний.** Если лендинг не опубликован — это видно сразу. Если счётчик не срабатывал 7 дней — об этом сказано в карточке.

## Информационная архитектура

### Верхнеуровневая навигация (левый сайдбар, сворачиваемый)

- Overview (сводка по портфелю)
- Projects (проекты → лендинги)
- Analytics (агрегированная аналитика)
- Forms (все формы и их заявки)
- Integrations (аналитика, CRM, мессенджеры, вебхуки)
- Domains (домены, SSL, редиректы)
- Activity (лог действий)
- Settings (команда, API-ключи, биллинг)

Внизу сайдбара — профиль, смена воркспейса, тема (light/dark), шорткаты (`⌘K`).

### Глобальные элементы

- **Command palette (`⌘K`)** — переход к любому проекту/лендингу, быстрые действия («опубликовать X», «сменить телефон в Y»), поиск по заявкам форм.
- **Глобальный поиск** в шапке — по проектам, лендингам, доменам, содержимому форм.
- **Нотификации** — колокольчик: новые заявки, упавшие счётчики, истечение SSL, ошибки деплоя.
- **Breadcrumbs** на всех внутренних страницах: `Projects / Клиент А / landing-black-friday`.

## Экраны — детально

### 1. Overview

Сводка владельца портфеля. Сверху — 4 KPI-карточки (без «пирожков» и 3D-графиков):

- Всего лендингов (опубликовано / черновики)
- Визиты за 7/30 дней + спарклайн
- Заявки из форм за 7/30 дней + спарклайн
- Средняя конверсия по портфелю + дельта к прошлому периоду

Ниже — две колонки:

- **Последние заявки** (живая лента из форм) — имя, телефон/почта, лендинг, время, статус (новая / просмотрена / в работе / закрыта).
- **Требует внимания** — умный список алертов: счётчик не отвечает, SSL истекает, форма не отправляется, деплой с ошибкой, резкое падение трафика. Каждая строка — ссылка на решение.

### 2. Projects (список проектов)

**Сетка карточек проектов**, 3 в ряд на десктопе. Переключатель «сетка / список».

Карточка проекта содержит:
- Название проекта, короткое описание
- Маленький favicon/логотип клиента (опционально)
- Бейдж статуса (активен / на паузе / архив)
- Счётчики: `12 лендингов` · `3 домена` · `4 формы`
- Мини-метрика: визиты за 7 дней (спарклайн) + дельта
- Последнее обновление: «изменён 2 часа назад»
- Меню действий (`•••`): открыть, переименовать, дублировать, архивировать

Сверху списка: поиск, фильтры (статус, клиент, тег), сортировка, кнопка **«Новый проект»**.

### 3. Карточка проекта (внутри — лендинги)

Шапка проекта: название, описание, теги, кнопки «Настройки проекта» и «Добавить лендинг».

Вкладки внутри проекта:
- **Landings** (основное)
- **Forms** (формы именно этого проекта)
- **Analytics**
- **Settings**

#### Сетка карточек лендингов

Каждая карточка лендинга — это основной рабочий объект продукта. Она должна быть максимально информативной без перегруза.

Структура карточки:
- **Превью** сверху — скриншот лендинга (генерируется автоматически при деплое), 16:9, с лёгкой рамкой.
- **Название** лендинга + slug/URL под ним (кликабельный, с иконкой копирования).
- **Статус-дот**: опубликован (зелёный) / черновик (серый) / ошибка (красный) / на проверке (жёлтый).
- **Мини-метрики в одну строку**: визиты · заявки · конверсия — за выбранный период (дефолт 7 дней).
- **Бейджи подключений**: какие счётчики подключены (GA, Яндекс.Метрика, VK Pixel, Meta Pixel, TikTok Pixel), какие формы активны, подключён ли CRM. Бейджи лаконичные, монохромные, с тултипом при наведении.
- **Строка быстрых действий** (появляется при hover или всегда видна внизу карточки):
  - Открыть редактор настроек
  - Превью (desktop / mobile переключатель)
  - Открыть опубликованную версию в новой вкладке
  - Меню `•••`: дублировать, архивировать, экспортировать HTML, история версий, удалить

Сверху списка лендингов: поиск, фильтры (статус, домен, наличие формы, период метрик), сортировка (по дате, по трафику, по конверсии), переключатель сетка/таблица, кнопка **«Добавить лендинг»** (импорт из Claude Code / загрузка HTML / создать пустой).

### 4. Экран управления одним лендингом

Это **центральный экран продукта**. Сюда пользователь приходит чаще всего. Разделён на левую панель-меню и основную рабочую область.

Шапка лендинга:
- Название, URL (с кнопкой копирования), статус
- Переключатель превью: Desktop / Tablet / Mobile
- Кнопки: **Preview** (в новой вкладке черновик), **Publish** (с указанием «будут опубликованы N изменений»)
- Индикатор несохранённых изменений
- Меню `•••`: история версий, откат, дублировать, экспорт, удалить

Левая панель секций настроек:

1. **General** — название, slug, заголовок вкладки (title), description, favicon, OG-image, OG-title, OG-description, canonical, robots.
2. **Contacts** — телефон (с маской и валидацией), email, адрес, часы работы, ссылки на мессенджеры (Telegram, WhatsApp). Изменения применяются ко всем полям на лендинге через переменные-плейсхолдеры.
3. **Analytics & Pixels** — подключение счётчиков: Google Analytics (GA4), Яндекс.Метрика (+ отдельный тумблер для вебвизора), Meta Pixel, VK Pixel, TikTok Pixel, Google Tag Manager, произвольный `<script>`-сниппет. Для каждого счётчика — поле ID, тумблер вкл/выкл, индикатор «последний хит N минут назад», кнопка «проверить установку».
4. **Forms** — список форм лендинга. Для каждой формы: название, поля (drag-n-drop, тип поля, обязательность, валидация, placeholder, маска), текст успеха/ошибки, защита от спама (honeypot, reCAPTCHA, rate-limit), назначение заявок (email, Telegram-бот, CRM, вебхук), редирект после отправки, события в счётчики.
5. **UTM & Traffic** — правила подмены контента по UTM (например, менять заголовок и телефон в зависимости от источника), список ожидаемых источников.
6. **A/B Tests** — запуск сплитов: вариант A/B, % трафика, целевое событие, сроки, текущие результаты со статзначимостью.
7. **Domain & SEO** — привязка домена/поддомена, статус SSL, редиректы (301/302), sitemap.xml, robots.txt, превью сниппета в Google.
8. **Integrations** — CRM (amoCRM, Bitrix24, HubSpot), почтовые сервисы, вебхуки, Zapier/Make.
9. **Code access** — просмотр исходного HTML с подсветкой синтаксиса (read-only по умолчанию, с кнопкой «открыть в Claude Code»), указание, какие блоки управляются дашбордом (подсветка зон).
10. **Advanced** — произвольные `<head>` и `<body>` инжекции, cookie-баннер, переадресации, пароль на доступ, индексация вкл/выкл.
11. **Danger zone** — архив, удаление.

Правая рабочая область — контент выбранной секции. Формы — в две колонки на широком экране, в одну на узком. Каждая настройка — с коротким хелпером под лейблом, без дублирующих тултипов.

Внизу экрана — **липкая панель** с действиями: «Сохранить черновик», «Опубликовать», «Отменить». Если есть несохранённые изменения — панель подсвечивается.

### 5. Forms (глобальный раздел)

Список всех заявок по портфелю — как инбокс.

- Слева — фильтры: по проекту, лендингу, форме, статусу, периоду, наличию тега.
- В центре — список заявок: имя, превью сообщения, источник (лендинг + UTM), время, статус.
- Справа — детали выбранной заявки: все поля формы, технические данные (IP, User-Agent, UTM, referrer), действия (сменить статус, назначить, добавить комментарий, экспорт, переслать).
- Массовые действия: экспорт в CSV, смена статуса, удаление.

### 6. Analytics

Агрегированная аналитика по портфелю с возможностью углубления до одного лендинга.

- Верхние KPI: визиты, уникальные, заявки, конверсия, bounce rate, среднее время.
- График визитов/заявок во времени с переключателем периода.
- Таблица «Топ лендингов» по трафику и по конверсии.
- Разбивка по источникам (UTM source/medium/campaign).
- Разбивка по устройствам и географии.
- Ссылки на внешние системы (GA, Метрика) для глубокой аналитики — не пытаться повторить их функциональность.

### 7. Integrations, Domains, Activity, Settings

- **Integrations**: каталог доступных интеграций с состоянием «подключено / не подключено», управление ключами, вебхуки с логом последних вызовов.
- **Domains**: список доменов и поддоменов, статус DNS, SSL, срок действия, привязка к лендингам, редиректы.
- **Activity**: лента действий команды (кто, что, когда, на каком объекте), фильтры, экспорт.
- **Settings**: воркспейс, команда и роли (owner / admin / editor / viewer), API-ключи, биллинг, уведомления.

## Состояния (не забыть)

Для каждого списочного и детального экрана продумать:

- **Empty state** — без проектов / без лендингов / без заявок / без интеграций. Содержательная иллюстрация-минимум, короткое объяснение, основной CTA и ссылка на доку.
- **Loading** — скелетоны, а не спиннеры по центру экрана.
- **Error** — что случилось, что сделать, кнопка «повторить».
- **Partial failure** — например, лендинг опубликован, но один счётчик не установился.
- **Permission denied** — для ролей viewer/editor.
- **Offline / устаревшие данные** — индикатор «данные от 12:45, обновить».

## Микровзаимодействия и UX-детали

- Inline-редактирование названий проектов и лендингов (клик по названию → поле → Enter).
- Копирование URL/ID — единая иконка-кнопка со статусом «скопировано» на 1.5с.
- Подтверждение деструктивных действий — модалка с вводом имени объекта (как в GitHub/Vercel).
- Undo-тост на 5 секунд после архивации/удаления, где это безопасно.
- Оптимистичные апдейты на смене статусов заявок, тумблерах, бейджах.
- Горячие клавиши: `⌘K` — палитра, `⌘/` — поиск, `G then P` — к проектам, `G then A` — к аналитике, `N` — создать, `E` — редактировать, `Esc` — закрыть панель.
- Drag-n-drop в редакторе полей формы и при сортировке лендингов внутри проекта.
- Автосохранение черновика настроек лендинга каждые N секунд, с явным индикатором.

## Визуальная система

- **Типографика**: один гротеск для интерфейса (например, Inter, Geist или Söhne-подобный), моноширинный для кода/ID/доменов (JetBrains Mono или Geist Mono). Чёткая шкала: 12 / 13 / 14 / 16 / 20 / 24 / 32.
- **Цвет**: нейтральная шкала серого (10+ ступеней), один акцентный цвет для основного действия и ссылок (сдержанный — глубокий синий, графит или чёрный как акцент на белом), семантические: успех (спокойный зелёный), внимание (янтарный), ошибка (приглушённый красный). Никаких фиолетово-розовых градиентов.
- **Темы**: light и dark, обе равноценны, не «затемнённый вариант» а самостоятельная палитра.
- **Сетка и ритм**: базовая единица 4px, основные отступы 16/24/32/48. Максимальная ширина контента — 1280–1440px, центрирование.
- **Компоненты**: плоские карточки с 1px бордером (не тень), радиусы 8–12px, едва заметные shadow только на hover и в модалках. Кнопки — три уровня (primary / secondary / ghost), один размер по умолчанию + sm/lg.
- **Иконки**: один набор на весь продукт (Lucide или аналог), линейные, 16/20/24px, без смешения стилей.
- **Данные**: таблицы с zebra отключена, разделители — тонкая линия, числовые колонки выравниваются по правому краю, моноширинный шрифт для чисел с фиксированной шириной цифр (`tabular-nums`).
- **Графики**: только линии и бары, без 3D, без пончиков. Один акцентный цвет + серый для сравнения.

## Адаптивность

- Основной сценарий — десктоп (1440+).
- От 1024 — сайдбар сворачивается в иконки.
- От 768 — одноколоночная раскладка, карточки на всю ширину, настройки лендинга — аккордеоном.
- Мобильная версия — полноценная, не «мобильный просмотр десктопа». Сфокусирована на просмотре метрик, работе с заявками и быстром включении/выключении настроек.

## Доступность

- Контраст минимум AA, для основного текста AAA.
- Все интерактивные элементы — фокус-стейт с видимым контуром.
- Логическая табуляция, skip-link на основной контент.
- ARIA-лейблы на иконочных кнопках.
- Поддержка `prefers-reduced-motion`.

## Что сдать (формат вывода)

1. **Схема информационной архитектуры** — карта разделов и связей.
2. **Карта экранов** — превью всех ключевых экранов на одном листе (flow).
3. **Высокоуровневые макеты** для:
   - Overview
   - Projects (сетка карточек)
   - Карточка проекта (список лендингов)
   - Экран управления одним лендингом — 3 состояния секций: General, Analytics & Pixels, Forms
   - Forms (инбокс заявок)
   - Analytics
   - Empty state и Error state для любого списочного экрана
4. **Компонентная библиотека** в миниатюре: кнопки, поля ввода, тумблеры, селекты, бейджи, карточки, таблица, модалка, тост, навигация.
5. **Описание логики**: что происходит при «опубликовать», при падении счётчика, при смене телефона (как это прокидывается во все лендинги проекта), при A/B-тесте.
6. **Спецификация визуальных токенов**: цвета, типографика, радиусы, тени, отступы — в виде таблицы.

## Критерии «хорошо сделано»

- Если показать макет маркетологу без разработчика — он понимает, где опубликовать лендинг и где сменить телефон, без инструкции.
- Если показать макет разработчику — он видит, где интегрировать Claude Code и как маппить настройки на HTML.
- В интерфейсе нет ни одного элемента, назначение которого нужно объяснять.
- Дизайн выглядит как продукт, которым пользуются 3 года, а не как концепт с Dribbble.

## Антипаттерны (чего не делать)

- Не использовать градиенты, неоны, glassmorphism, glow-эффекты.
- Не делать «AI-sparkle» иконки и «magic»-копирайт в интерфейсе.
- Не перегружать карточки: максимум 5–6 сущностей информации на карточку лендинга.
- Не прятать критические действия («опубликовать», «сменить телефон») глубже одного клика от карточки.
- Не повторять функциональность Google Analytics/Метрики — только агрегировать и давать ссылки.
- Не делать онбординг-туры поверх продукта — лучше хорошие empty states.
